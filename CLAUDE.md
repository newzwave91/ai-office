## 오케스트레이션 (헤르메스 방식)
사용자 요청을 받으면 먼저 성격을 분류해 담당 팀장(서브에이전트)에게 위임한다.

분류 키워드 (매칭되는 한 명에게 우선 위임):
- 마케팅 / 콘텐츠 / 홍보 / 채널 운영 / 광고·후크 문구 → marketing (Echo)
- 사업전략 / 매출 / KPI / 시장·경쟁 분석 / 가격·수익 모델 → strategy (Oracle)
- 회의록 / 업무보고 / 문서 정리 / 프로세스·온보딩 설계 → ops (Link)
- 계약·문구 검수 / 규정·법규 / 리스크·금지표현 점검 → review (Glitch)
- 팀 전체 업무 취합 / 우선순위 조율 / 주간 정리 / 업무 최적화 → director (Neo)

최소 동원 원칙 (반드시 준수):
- 단순 조회·확인·단답 요청은 담당 팀장 1명만 호출한다. 관련 없는 팀장을 끌어오지 않는다.
- 창작·기획·전략처럼 여러 직무가 실제로 필요할 때만 2~3명을 동원한다. 한 요청에 4명 이상 동시 동원은 금지 — 팀장 회의가 필요하면 /회의 명령을 쓴다.
- 사용자가 "검수"·"디자인"·"매출" 같은 특정 직무 단어를 직접 쓰지 않았다면 그 직무 팀장을 임의로 추가하지 않는다. 데이터·사실 확인 없이 분석 팀장만 호출해 추측성 분석을 내놓지 않는다.

여러 직무에 걸치는 요청: 주 담당 한 명을 정해 위임하고, 그 산출물을 다른 팀장 관점으로 교차 검토한다. 작업을 분해할 때는 먼저 "이번 일이 무엇인지 2~3줄 요약(brief) → 팀장별 구체·실행가능 임무(tasks)" 순으로 정리한 뒤, 논리적 순서(데이터·사실 확인 → 분석 → 창작)로 실행한다.

분류가 불확실하면 추측하지 말고 사용자에게 한 번 묻는다.

## 지식 금고 (자동 메모리)
- 대화 중 보존 가치가 있는 정보(고객 사실, 결정사항, 사용자 선호·규칙)를 발견하면, 작업 완료 후 vault/00_지식금고/ 아래 해당 분야 폴더에 마크다운 노트로 기록한다.
- 노트 형식: 파일명은 "주제 키워드.md". 첫 줄에 YAML frontmatter(date, source, tags)를 쓰고, 관련 개념은 [[위키링크]]로 연결한다. 한 노트 = 한 가지 사실, 3문장 이내.
- 같은 내용의 노트가 이미 있으면 새로 만들지 말고 기존 노트를 갱신한다.
- 잡담, 일회성 질문, 추측은 기록하지 않는다.
- 작업 시작 전에는 관련 폴더의 노트를 먼저 검색해 읽고, 금고 내용과 모순되는 답을 내지 않는다.
- 결정 로그: 작업·대화에서 "앞으로 회사가 따를 명시적 결정·원칙"이 나오면 vault/00_지식금고/공통/결정로그.md에 append-only로 적립한다. 형식은 "- [YYYY-MM-DD] 결정 내용"(명령형/단정형, 건당 60자 이내). 추측·일반론·사담은 제외하고, 한 작업에서 0~3개만 뽑으며 없으면 적지 않는다. (원본 decisions-extract 패턴)
- 스킬 증류(카파시 스킬): 완성된 산출물에서 재사용 가능한 패턴·노하우가 보이면 /스킬 명령으로 vault/00_지식금고/스킬/ 폴더에 "스킬 - 제목.md"(tags에 스킬 포함)로 적립한다. 스킬 노트 형식은 첫 줄 frontmatter + `# 스킬 - 제목` + `_언제 쓰나: …_` + 핵심 패턴/단계/체크리스트/예시. 검증된 것만 쓰고 구체 수치·표현은 그대로 보존하며, 재사용 가치가 모호하면 적립하지 않는다. 적립하면 vault/00_지식금고/스킬/스킬 인덱스.md(노하우 카탈로그)도 함께 갱신한다. 사실(지식금고 노트)과 노하우(스킬 노트)는 구분한다 — 사실은 분야 폴더, 노하우는 스킬 폴더.
- 대화 로그: 대표의 모든 대화(질문·말투·표현)는 vault/대표_대화로그.md에 서버가 자동 누적한다(지식금고 밖, 불변 원본 성격). 이 파일은 대표가 자신의 말투·질문 패턴 분석 등을 명시적으로 요청할 때만 읽어 분석하고, 평상시 작업·답변·금고 검색에는 절대 참조하지 않는다. 분석 결과 중 보존 가치가 있으면 vault/00_지식금고/공통/에 노트로 적립한다.

## 지식 위키 (LLM Wiki 운영)
- 3계층 구조를 유지한다: ① vault/01_원문/ + vault/02_외부자료/ = 불변 원본(절대 수정·삭제 금지, 읽기만), ② vault/00_지식금고/ = LLM이 전적으로 관리하는 위키, ③ CLAUDE.md = 스키마(이 문서). 위키(②)는 두 종류의 노트를 담는다 — 분야 폴더의 **사실 노트**(LLM 위키)와 스킬 폴더의 **노하우 노트**(카파시 스킬). 전체 입구는 vault/🏠 홈.md(옵시디언 홈 MOC)다.
- 02_외부자료/는 외부에서 받은 자료(github, claude-code, 트렌드 하위 분류)를 모으는 곳이다. 새 자료가 들어오면 /수집으로 핵심을 지식금고에 증류한다.
- 수집(ingest): 새 원문이 들어오면 vault/01_원문/에 "YYYY-MM-DD 제목.md"로 저장한 뒤, 핵심을 추출해 지식금고의 관련 노트들에 통합한다(갱신 우선, 필요 시 신규 생성, [[위키링크]] 연결). 원문 하나가 노트 여러 개를 건드릴 수 있다.
- 모순 처리: 새 정보가 기존 노트와 충돌하면 임의로 덮어쓰지 말고, 해당 노트에 "[모순 YYYY-MM-DD] 내용" 줄을 남기고 사용자에게 보고한다.
- index.md / 스킬 인덱스.md: vault/00_지식금고/index.md는 **사실 노트** 카탈로그([[링크]] + 한 줄 요약, 분야별 분류), vault/00_지식금고/스킬/스킬 인덱스.md는 **노하우(스킬)** 카탈로그다. 사실 노트를 만들거나 갱신하면 index.md를, 스킬 노트면 스킬 인덱스.md를 함께 갱신한다. 작업 시작 전 검색은 index.md(필요하면 스킬 인덱스.md)부터 읽는다.
- 템플릿: 새 노트는 vault/_템플릿/의 양식(지식 노트·스킬 노트·산출물·회의록)을 따른다. _템플릿/은 옵시디언 Templates 폴더이며, 앱 탐색기·자동 적립 대상이 아니다(형식 참고용).
- log.md: vault/00_지식금고/log.md는 append-only 기록이다. 형식: "## [YYYY-MM-DD] 유형 | 제목" (유형: 수집/기록/회의/질의/점검/결정/스킬). 기존 줄은 수정하지 않는다.
- 질의 환류: 대화에서 나온 가치 있는 분석·비교·결론은 채팅에 흘려보내지 말고 노트(사실)나 산출물(문서)로 적립한다.
- 점검(lint): /점검 실행 시 모순, 중복, 고아 노트(들어오는 링크 없음), 깨진 위키링크, index와 실제 파일 불일치, 낡은 정보를 검사한다.

## 산출물 규칙
- 모든 결과 문서는 vault/90_산출물/에 저장한다. 정리된 회의록도 완성 문서이므로 90_산출물/에 저장하며, 10_회의록/은 (선택) 회의 원본 메모를 두는 곳이다.
- 업종 특수 규칙(법규, 금지 표현, 브랜드 톤 등)이 지식 금고의 "공통" 폴더에 있으면 모든 작업에서 최우선으로 지킨다.
